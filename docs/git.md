# Here is some of the git rulessets we follow in this project

## Branching
### Naming
When creating new branches we use Purpose Prefixes — describe the intent of the work:

- <b>feature/ (or feat/):</b> For new features (e.g., feature/add-login-page, feat/add-login-page)
- <b>bugfix/ (or fix/):</b> For bug fixes (e.g., bugfix/fix-header-bug, fix/header-bug)
- <b>hotfix/:</b> For urgent fixes (e.g., hotfix/security-patch)
- <b>release/:</b> For branches preparing a release (e.g., release/v1.2.0)
- <b>chore/:</b> For non-code tasks like dependency, docs updates (e.g., chore/update-dependencies)


## Issues
When ai is asked to create an issue,, its important to follow the rules below:
- Use a clear and descriptive title that summarizes the issue.
- If Possible, then provide useful links to relevant design or documentation resources.
- Make sure to always add tags to the issue to make it easier to find and categorize issues.


## Pull requests
- Make sure to link the affected issue in the pull request description, so that it will also be closed when the pull request is merged.
